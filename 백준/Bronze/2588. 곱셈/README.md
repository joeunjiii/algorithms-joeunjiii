### 1) `BufferedReader.readLine()`이 하는 일
- **한 줄 전체를 문자열(String)로 읽어옵니다.**
- 줄바꿈(`\n`)은 제외된 상태로 반환됩니다.
- **더 이상 읽을 줄이 없으면 `null`을 반환**합니다.
  - 이 상태에서 `Integer.parseInt(null)`을 하면 런타임 에러가 납니다.
```java
예시 입력:
472
385
```

markdown
코드 복사
`readLine()` 호출 결과:
- 1번째: `"472"`
- 2번째: `"385"`
- 3번째: `null`  ← 여기서 파싱하면 터짐

---

### 2) `StringTokenizer`가 하는 일
- `readLine()`으로 읽은 **한 줄 문자열을 공백(기본) 기준으로 토큰(단어)로 분리**합니다.
- `nextToken()`으로 토큰을 하나씩 꺼냅니다.
- **토큰이 없는데 `nextToken()`을 호출하면 런타임 에러**(`NoSuchElementException`)가 납니다.

예시:
- `"10 20 30"` → 토큰 3개
- `"472"` → 토큰 1개 (여기서 `nextToken()`을 2번 호출하면 2번째에서 터짐)

---

### 3) 언제 무엇을 써야 하나? (가장 중요한 기준)

#### A. 입력이 “줄마다 값 1개”인 경우 (예: 2588번)
- 토크나이저가 **필요 없습니다.**
- `readLine()`을 줄 수만큼 호출해서 바로 파싱합니다.
- 핵심: **입력 줄 수만큼만 읽기** (불필요하게 한 번 더 읽지 않기)

#### B. 입력이 “한 줄에 여러 값(공백 구분)”인 경우 (예: 10430번)
- `readLine()` 1번으로 한 줄을 읽고,
- `StringTokenizer`로 공백 기준 분리 후 `nextToken()`으로 꺼냅니다.
- 핵심: **토큰 개수만큼만 `nextToken()` 호출하기**

---

### 4) 런타임 에러가 자주 나는 패턴
- `readLine()`을 입력 줄 수보다 더 많이 호출해서 **`null`을 파싱**
- 한 줄에 토큰이 1개인데 `nextToken()`을 2개 이상 꺼내려 해서 **토큰 부족**

---

### 5) 결론
- `readLine()` = “한 줄 통째로 읽기”
- `StringTokenizer` = “한 줄을 공백으로 쪼개기”
- 입력 형식에 맞춰서:
  - **두 줄 입력(값 1개씩)** → `readLine()` 2번
  - **한 줄 입력(값 여러 개)** → `readLine()` 1번 + `StringTokenizer`
